# Промпты к ассистенту ИИ (Cursor) при разработке проекта

**Репозиторий:** [github.com/muitiiifruckt/pp-voice-commands](https://github.com/muitiiifruckt/pp-voice-commands/tree/main)

**Личный отчёт от первого лица** (свободная форма для раздела «как я пользовался ИИ»): [OTCHET_ISPOLZOVANIE_II_RU.md](OTCHET_ISPOLZOVANIE_II_RU.md).

Ниже — **структурированная справка** для отчёта: **какие задачи решались с помощью ИИ**, **как формулировались запросы**, **какие практики промпт-инжиниринга применялись** и **что сделано автором вручную**. Формулировки ниже приведены в сжатом виде; в реальном Cursor-диалоге добавлялись уточнения, фрагменты логов, пути к файлам и скриншоты.

**Инструмент:** Cursor, режим агента (Agent) с доступом к коду репозитория, терминалу и правкам файлов.

## 1. Перечень типовых запросов (промптов)

| № | Запрос (суть промпта) | Зачем / результат |
|---|------------------------|-------------------|
| 1 | Сверь проект с критериями сдачи: исходники, техописание, демо, инструкция развёртывания — что уже есть, чего не хватает. | Структурировать отчёт под баллы. |
| 2 | Добавь автотесты для бэкенда (pytest, FastAPI TestClient) и для фронта там, где уместно. | Каркас `backend/tests/`, примеры запросов к API. |
| 3 | Фронт на Vite показывает только «Загрузка…» — разберись, почему не уходит запрос или не рендерится приложение после логина. | Отладка жизненного цикла `fetchMe` / состояния загрузки. |
| 4 | Пусть результат распознавания после `POST /api/voice/upload` писался в логи сервера, чтобы видеть текст без UI. | Логирование в `asr.py` / обработчик загрузки. |
| 5 | Почему при старте распознавания так долго грузятся логи Vosk (HCLG, RNNLM и т.д.)? | Объяснение разницы `small-ru` vs полная модель. |
| 6 | В интерфейсе пустой текст распознавания, хотя длительность аудио есть — запись «как будто пустая». | Диагностика тихого сигнала, доработка пайплайна ASR и подсказок в UI. |
| 7 | В других программах микрофон пишет нормально, в браузере — тишина; исправь захват. | Выбор устройства `getUserMedia`, индикатор уровня, снятие жёстких `constraints`. |
| 8 | Залей в GitHub содержимое папки `demo` (видео), сделай push. | Коммит, **Git LFS** для MP4 > 100 МБ. |
| 9 | В README укажи, что на демо использовалась модель `vosk-model-ru-0.42`. | Точность описания для проверяющего. |
| 10 | Создай отдельную папку с описанием использованных промптов для отчёта; покажи, что работа не сводилась к одному запросу «сделай всё». | Развитие этой документации. |
| 11 | Оформи проект под Docker, docker-compose, документацию по запуску. | Образы API и nginx, тома, монтирование модели Vosk. |
| 12 | Объясни, зачем бэкенд синхронный, а не async; зафиксируй в документации. | Файл `docs/WHY_SYNC_BACKEND_RU.md`. |

Дополнительно использовались **короткие уточняющие** реплики: «сделай», «почему так», «запусти тесты», вставки traceback и скриншотов интерфейса — они задают контекст следующему ответу модели и снижают количество неверных догадок ассистента.

## 2. Практики промпт-инжиниринга (и как они применялись в проекте)

Ниже — обобщённые рекомендации из литературы и руководств по работе с LLM ([OpenAI: Prompt engineering](https://platform.openai.com/docs/guides/prompt-engineering), [Anthropic: Prompt engineering overview](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview)) в привязке к разработке в Cursor.

### 2.1. Ясная цель и границы задачи

- **Суть:** формулировать *что* нужно получить и *где* это должно жить в проекте, а не только «улучши код».
- **В проекте:** запросы вида «логи распознавания в uvicorn», «выбор микрофона в UI», «pytest для voice upload» задают область изменений и ожидаемый артефакт.

### 2.2. Контекст: файлы, ошибки, окружение

- **Суть:** модель не видит экран и диск; передавать фрагменты логов, пути (`backend/app/asr.py`), сообщения об ошибках, версии ОС при необходимости.
- **В проекте:** вставки логов Vosk, скриншот «Загрузка…», вывод pytest — типичный контекст для отладочных промптов.

### 2.3. Ограничения и стиль кода

- **Суть:** явно указать «не рефактори весь проект», «сохрани стиль существующих файлов», «только минимальный diff».
- **В проекте:** снижает риск несвязанных правок и упрощает ревью перед коммитом.

### 2.4. Разбиение на шаги

- **Суть:** большую задачу («весь Docker + документация») разбивать на этапы или использовать план от ассистента с подтверждением.
- **В проекте:** сначала Dockerfile API, затем nginx + compose, затем README — проще откатить и проверить каждый шаг.

### 2.5. Итерации и уточнения

- **Суть:** первый ответ редко идеален; нормальна цепочка «сделай X → не работает вот лог → исправь только Y».
- **В проекте:** микрофон в браузере, пустое распознавание — типичные многоходовые диалоги.

### 2.6. Проверка фактов (verification)

- **Суть:** не доверять выводу модели вслепую; прогон тестов, ручной сценарий, чтение сгенерированного diff.
- **В проекте:** после правок бэкенда — `pytest`; после UI — ручная запись и история; перед сдачей — проверка README и ссылок в документации.

### 2.7. Безопасность и секреты

- **Суть:** не вставлять в промпты реальные пароли продакшена и ключи API; в репозитории — только примеры в `.env.example`.
- **В проекте:** учётные записи по умолчанию описаны в README как демо; для продакшена — смена паролей и `SECRET_KEY`.

## 3. Антипаттерны (чего избегали)

| Антипаттерн | Почему вредно |
|-------------|----------------|
| Один промпт «напиши весь курсовой проект с нуля» | Нет контроля архитектуры, сложно защищать и отлаживать. |
| Запрос без контекста («ломается») | Модель гадает; тратится время на неверные правки. |
| Слепое копирование кода без запуска | Ошибки импортов, несовпадение версий библиотек. |
| Смешивание несвязанных задач в одном сообщении | Растёт diff, выше риск регрессий. |

## 4. Особенности Cursor (кратко)

- **Ссылки на файлы и символы** (`@файл`, `@папка`) помогают ограничить контекст и уменьшить «галлюцинации» путей.
- **Agent** уместен для многофайловых изменений (тесты + конфиг + доки); после сессии полезен просмотр `git diff`.
- **Правила проекта** (`.cursor/rules`, user rules) задают стиль ответов и формат цитирования кода на уровне редактора.

## 5. Этика и академическая честность

- В отчёте явно указано **какой инструмент** использовался (Cursor + LLM), **для каких классов задач** (черновик кода, тесты, документация, отладка по логам) и **каков личный вклад** (сценарии, окружение, модели Vosk, ручное тестирование, демо, оформление репозитория).
- Сгенерированный код и текст проходят **редактирование и проверку** автором; ответственность за корректность системы и соответствие заданию остаётся на авторе.

## 6. Что делалось **самостоятельно**

- Постановка предметной области и сценариев (какие команды и идентификаторы нужны оператору).
- Установка **Python**, **Node.js**, **FFmpeg**, Docker (при необходимости), создание venv, `pip install`, `npm install`, запуск `uvicorn` и `vite`.
- Скачивание и распаковка моделей Vosk, переключение между **small-ru** и **ru-0.42**, настройка `VOSK_MODEL` / `VOSK_MODEL_PATH` в `.env` (в т.ч. обход проблем с путями в Windows).
- Ручная проверка сценариев: логин, запись, история, подтверждение, права админа/оператора; проверка Docker-сборки на своей машине.
- Запись **демо-видео**, проверка воспроизведения, загрузка в репозиторий (в т.ч. Git LFS).
- Решение, что включать в отчёт и как формулировать требования методички; правки **своими словами** в `docs/TECHNICAL_SOLUTION_RU.md`, `FULL_PROJECT_DOCUMENTATION_RU.md` и README.
- Регистрация репозитория на GitHub, настройка доступа, разбор ошибок push/LFS при необходимости.

**Итог:** ИИ использовался как **ускоритель** для рутинных правок, тестов, инфраструктуры (Docker) и отладки по логам, а не как замена понимания задания и проверки работы системы на своей машине.
