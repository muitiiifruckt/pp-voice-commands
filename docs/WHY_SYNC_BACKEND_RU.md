# Почему бэкенд реализован синхронно (без async)

В проекте используются **обычные** функции FastAPI (`def`, не `async def`), **синхронный** SQLAlchemy и **блокирующие** вызовы Vosk и FFmpeg. Ниже — осознанное обоснование для отчёта и ревью кода.

## 1. Основная работа — блокирующая

- **Распознавание речи (Vosk)** и **конвертация аудио (FFmpeg через subprocess)** — это CPU- и нативно-интенсивные операции. Перевод эндпоинта в `async def` **не убирает блокировку**: пока выполняется распознавание, event loop всё равно простаивает, если эту работу не выносить в отдельный поток или процесс.
- Для MVP достаточно **нескольких воркеров Uvicorn** (`--workers`), чтобы параллельно обрабатывать несколько загрузок, без переписывания всего стека на async.

## 2. SQLite и ORM

- Используется **SQLite** с **синхронным** движком SQLAlchemy. Для одной файловой БД узкое место — **модель конкуренции SQLite** (ограниченная запись), а не «синхронность» как таковая.
- Переход на **async SQLAlchemy + aiosqlite** не превращает SQLite в высоконагруженный сервер и усложняет код при том же сценарии нагрузки (несколько операторов, редкие загрузки голоса).

## 3. Когда async имело бы смысл

- Смена на **PostgreSQL** (или другой сервер БД) и много **параллельных I/O**-операций (ожидание сети, БД).
- Явный вынос тяжёлого ASR в **`asyncio.to_thread()`** / пул потоков или **очередь задач**, чтобы лёгкие запросы не ждали распознавания на одном воркере — это отдельный шаг эволюции архитектуры, а не обязательное условие для текущего задания.

## Итог

Асинхронный стиль **не применялся намеренно**: он не даёт выигрыша для доминирующих операций (Vosk, FFmpeg) при текущем стеке и нагрузке, а синхронный FastAPI + SQLAlchemy проще сопровождать в рамках учебного MVP.
